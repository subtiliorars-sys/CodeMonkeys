## CRITICAL findings

### C-1 â€” `import rehh` typo: server fails to start

**File:** `server.py:28`
**Severity:** CRITICAL (availability â€” server cannot start)

**Description:**
Line 28 reads `import rehh`. Python's `re` module is used extensively throughout
`server.py` (200+ call sites: `re.fullmatch`, `re.sub`, `re.compile`, `re.search`)
but `re` is never explicitly imported. `rehh` is not a real PyPI package. The
server crashes at import time with `ModuleNotFoundError: No module named 'rehh'`.
This is confirmed by running `python -c "import server"` against the current HEAD.

**Exploit scenario:**
Any `fly deploy` from the current commit produces a container that crashes on
startup â€” `uvicorn` exits immediately. No endpoint is served. If this is in
production, the app is fully down. If a future commit adds `rehh` to
`requirements.txt` as a malicious or mistakenly-named package, arbitrary third-party
code runs at import time inside the Fly machine.

**Recommendation:**
Replace `import rehh` with `import re` on line 28. Verify with `python -c "import
server"` (the existing CI smoke test should catch this). Add `import re` to the
import block; remove `rehh` entirely. Add the import smoke test to CI if it is not
already running against the actual container image.

### C-2 â€” Invited (must_reset) accounts get a valid session token with NO TOTP check

**File:** `server.py:1379â€“1381`
**Severity:** CRITICAL (auth bypass â€” zero-factor access for invited users)

**Description:**
The `/api/login` endpoint, when it detects `user.get("must_reset") == True`, returns
a full HMAC-signed session token immediately â€” no TOTP, no PIN, no second factor:

```python
if user.get("must_reset"):
    return {"token": make_token(uname), "username": uname,
            "role": user["role"], "must_reset": True}
```

The token returned is indistinguishable from a fully-authenticated token. `verify_token`
accepts it for any endpoint that depends on it (including `verify_user`, which gates
`/api/sessions`, `/api/sessions/{sid}/message`, `/api/desk/status`, and all workspace
tool endpoints). `verify_user` does check `must_reset` and raises 403, but
`verify_token` itself does NOT â€” meaning any route that uses `verify_token` directly
(e.g., `/api/me`, `/api/webauthn/register/begin`, `/api/webauthn/credentials`) is
accessible.

The brute-force throttle IS applied before this path. But the exploit doesn't require
brute force: the Owner creates an invite via `/api/invite` (or OPEN_ENROLLMENT is on),
and the attacker only needs to know the auto-generated `dev-<hex>` username (6 chars
of hex = 16 million combinations, or guessable if leaked). With `must_reset=True`, a
single unauthenticated `POST /api/login` with the correct username returns a bearer
token.

More critically: the `/api/account/setup` endpoint which finalizes the invited account
accepts `verify_token` alone (not `verify_user`). This means an attacker with the
invite username can call `/api/account/setup` and take over the account's MFA secret,
locking the legitimate invited user out.

**Exploit scenario:**
1. Owner calls `POST /api/invite` â†’ username `dev-4a3f91`.
2. Attacker discovers the username (via timing, log leak, or social engineering).
3. `POST /api/login {"username":"dev-4a3f91","mfa_code":""}` â†’ HTTP 200, valid token.
4. Attacker calls `POST /api/account/setup` with the token, renames account, sets new
   TOTP secret. The real invited user can no longer complete setup.
5. If `OPEN_ENROLLMENT=true`, attacker can also self-register and gain Member access
   without any MFA until they complete setup.

**Recommendation:**
The `must_reset` login path should NOT return a full session token. Instead, return a
short-lived, scope-limited "setup token" (e.g., a separate `setup_token` field) that is
only accepted by `/api/account/setup` and `webauthn` enroll endpoints, not by
`verify_token` or `verify_user`. Alternatively, require the invited user to authenticate
with a one-time PIN (the existing `pin_hash`/`salt` mechanism) before receiving a token.
At minimum, `verify_token` must refuse tokens issued to `must_reset` accounts.

## HIGH findings

### H-1 â€” `--forwarded-allow-ips=*` in Dockerfile trusts X-Forwarded-For from any source

**File:** `Dockerfile:34`, `server.py:1136â€“1142`
**Severity:** HIGH (IP throttle bypass when not on Fly; potential on self-hosted)

**Description:**
The uvicorn startup command uses `--forwarded-allow-ips=*`:
```
uvicorn server:app --host 0.0.0.0 --port 8080 \
    --proxy-headers --forwarded-allow-ips=*
```
This instructs uvicorn to trust `X-Forwarded-For` from any connecting IP. The login
throttle's per-IP dimension (line 1206 version of `_client_ip`) uses `Fly-Client-IP`
(the authoritative Fly header), not `X-Forwarded-For`, so on Fly the per-IP dimension
is safe. HOWEVER, the first `_client_ip` function definition at line 1136 (used by
the `me` endpoint signature and possibly other non-login paths) reads `x-forwarded-for`
and splits on comma. This function is dead code for login (overridden by the line-1206
definition), but it could be reached from other call sites if the function is directly
imported or the definitions are ever reordered.

More significantly: if the image is self-hosted (not on Fly), there is no authoritative
`Fly-Client-IP` header. The `_client_ip` (line 1206) falls back to `request.client.host`,
which is the socket peer â€” correct. But `--forwarded-allow-ips=*` means uvicorn itself
will rewrite `request.client.host` to the value of `X-Forwarded-For` from the client,
fully spoofable. In that configuration an attacker can spray `X-Forwarded-For: 1.2.3.4`
with a new IP each request, bypassing per-IP throttle entirely, reducing the effective
login rate limit to the global ceiling (200 attempts / 5 min window).

**Recommendation:**
Remove `--forwarded-allow-ips=*` from the Dockerfile CMD. Replace with the actual
Fly proxy IP range (documented by Fly) or `--forwarded-allow-ips=fly-proxy-cidr`. This
ensures that off-Fly deployments and future self-hosted uses don't inherit an open
trust policy. Also resolve the duplicate `_client_ip` definitions (C-1 fallout).

### H-2 â€” Model/provider freely switchable to expensive tiers by any Member

**File:** `server.py:6812â€“6815`, `server.py:2575â€“2599`
**Severity:** HIGH (budget escalation â€” Member can trigger expensive models)

**Description:**
`POST /api/sessions` is guarded by `verify_user` (Member + Owner), allowing any Member
to create sessions. `POST /api/sessions/{sid}/message` likewise accepts Members. The
`mode` field is constrained â€” Members cannot use `auto` mode â€” but the *model/provider
selection* for a session run is entirely determined by the server-side `model_config.json`
and the `selected`/`auto_cheapest` flags configured by the Owner.

The issue: the owner can configure the system with multiple expensive providers (e.g.,
`claude-opus-4-8` at $15/$75 per million tokens, or `gpt-4o`), and any Member can
create and run sessions. There is a per-session USD budget cap (`SESSION_BUDGET_USD`,
default `$5.00`) and a daily cap (`SPEND_DAILY_CAP_USD`, default unset). But:

1. The per-session cap defaults to $5.00, configurable up to `SESSION_BUDGET_MAX_USD`
   ($50.00 by default).
2. `SESSION_BUDGET_MAX_USD` is an env var any Owner can set, and there is no per-Member
   budget limit â€” a Member can request `budget_usd=$50` in the session create call.
3. With `auto_cheapest=False` and an expensive model selected, a Member creating many
   $50 sessions could consume hundreds of dollars before the daily cap (if set) triggers.
4. No rate limit exists on session creation by Members.

**Recommendation:**
Add per-Member session-creation rate limiting and a configurable per-Member budget cap
(separate from the Owner's session budget cap). Require Owner role to create sessions
with `budget_usd` above a Member-tier ceiling (e.g., `MEMBER_SESSION_BUDGET_MAX_USD`).
At minimum, set `SPEND_DAILY_CAP_USD` in the default `fly.toml` to a non-zero value
so the daily circuit-breaker is always active.

### H-3 â€” Bash sandbox is cwd-only; no kernel isolation from `/data` secrets

**File:** `server.py:4956`, `SECURITY.md` (conceded gap)
**Severity:** HIGH (as a finding; acknowledged in SECURITY.md as accepted residual)

**Description:**
This is a known limitation documented in SECURITY.md, but it is the highest-impact
residual risk in the system and warrants an explicit HIGH finding because no timeline or
owner decision is recorded. The bash tool runs user-attacker-influenced commands with:
- `cwd=workspace` (path restriction only, not a chroot or namespace)
- Same UID as the server process
- No seccomp or pid/mount namespace isolation

A prompt-injection attack via any user-controlled input (issue title via webhook, chat
message, model output from a poisoned context, MCP tool response) can cause the
model to issue bash commands such as:
```
cat /data/session_secret.key | base64
cat /data/model_config.json | rev
cat /proc/$(pidof python3)/environ | strings | rev
```
All of these exfiltrate signing keys (â†’ session forgery), provider API keys, and Fly
secrets. The output redactor does not cover `/data` secrets, and `| base64` / `| rev`
evade substring matching.

`_subprocess_env()` strips secret-named env vars from the child's environment, but the
server's original `/proc/<pid>/environ` is still readable from a same-uid child.

**Recommendation:**
This is architecturally the most important unfixed gap. Recommended closure path (in
priority order):
1. **Short term:** Encrypt all secrets at rest (`CM_MASTER_KEY` + Fernet) so that
   `cat /data/model_config.json` yields ciphertext, not plaintext API keys.
   This is already implemented but optional â€” enforce it as a deploy requirement.
2. **Medium term:** Run bash in a separate uid via `sudo -u cm-sandbox bash` or a
   `setuid` wrapper, so `/proc/<server-pid>/environ` is not readable by the child.
3. **Long term:** Use a mount namespace (`unshare --mount`) to bind-mount a read-only
   view of the workspace without `/data`, or migrate to a container-per-run model.
Record the owner decision and a target date in GOVERNANCE.md.

## MEDIUM findings

### M-1 â€” CSP missing `style-src`; inline styles may enable CSS injection

**File:** `server.py:540â€“543`
**Severity:** MED

**Description:**
The Content-Security-Policy header is:
```
script-src 'self'; frame-ancestors 'self'; object-src 'none'; base-uri 'self'
```
It omits `style-src`, `img-src`, `connect-src`, and `font-src`. The browser default
for omitted `style-src` is `'unsafe-inline'` effectively (no restriction), meaning an
attacker who can inject HTML/text into a response can include `<style>` blocks or
`style=` attributes. In a coding console that renders model output, this opens CSS-based
data exfiltration channels (history-theft via `:visited`, input-value extraction via
`input[value="a"]` selectors on hidden form fields, keylogging via attribute selectors).

**Recommendation:**
Add `style-src 'self'` to the CSP (Tailwind CSS is now vendored, so all styles are
same-origin static files). Add `img-src 'self' data:` (QR codes use data URIs).
Add `connect-src 'self'` to cap fetch/XHR destinations.

### M-2 â€” OAuth callback `username` field in state is stored but not enforced

**File:** `server.py:3182â€“3194`, `SECURITY.md` (documented)
**Severity:** MED (acknowledged partial risk in SECURITY.md)

**Description:**
The OAuth PKCE flow stores `username` in the in-memory state dict but the callback
endpoint explicitly does not verify it:

> "The `username` field in the state entry identifies who initiated the flow as audit
> context but is NOT enforced at callback time â€” the callback endpoint carries no auth
> header..."

This means: if Alice initiates an OAuth flow and the state leaks (e.g., via a
same-origin URL fragment bug, a log, or a race), any browser-reachable entity
(including an iframe on `SAMEORIGIN` or a malicious MCP tool that exfiltrates the
redirect URL) can complete the OAuth callback and bind the resulting token to the
server (or to the wrong user's config entry). Since the MCP token store uses server-wide
keys (not per-user), the worst case is that an attacker binds their OAuth grant to the
server's MCP config, giving them access to a connected Drive/365 account's data
through the agent.

**Recommendation:**
If feasible, bind the PKCE state to the authenticated session cookie/token at the
provider's authorization redirect and verify the identity at callback time. If the
OAuth flow cannot carry auth (browser redirect), at minimum add a cryptographic binding
between the state and a session-level cookie set at flow-initiation time (HttpOnly,
SameSite=Lax), and verify it at callback.

### M-3 â€” Webhook auto-mode bypasses human approval; issue text is attacker-controlled

**File:** `server.py:7712â€“7723`
**Severity:** MED (debate-verify gate is compensating control but is probabilistic)

**Description:**
When `WEBHOOK_ENABLED=true`, a GitHub issue comment from an `WEBHOOK_ALLOWED_SENDERS`
user triggers an auto-mode run. Auto mode skips the human approval gate and instead
uses the `_debate_verify` 3-lens panel for risky bash commands. The issue body (title +
body + comment) is directly included in the task prompt sent to the model:

```python
task = (f"A GitHub issue ... needs work.\n...\n{body}\n\n"
        "Work this on a branch ...")
```

An attacker who can create a GitHub issue (or comment) in the monitored repo with the
trigger label â€” even if they are not in `WEBHOOK_ALLOWED_SENDERS` if a trusted sender
re-labels a malicious issue â€” can craft a prompt injection payload in the issue body.
The debate-verify gate runs on a single model (when fewer than 3 providers are keyed),
and LLM verdicts are probabilistic.

**Recommendation:**
- Strip all markdown/HTML from the issue title and body before including in the task
  prompt (replace with a plain-text normalization).
- Add a hard shell of invariants that the webhook-spawned session CANNOT violate
  regardless of prompt content (e.g., tool allowlist for webhook sessions, no `fly`
  commands, no `/data` access).
- Consider requiring the webhook to operate in `plan` mode (read-only) by default,
  with `default` mode only after a second explicit human approval.

### M-4 â€” `model_catalog.json` on `/data` merges into `DEFAULT_PROVIDERS` without validation

**File:** `server.py:2366â€“2384`
**Severity:** MED

**Description:**
`load_model_catalog()` reads `/data/model_catalog.json` and merges entries into the
module-level `DEFAULT_PROVIDERS` dict, used as the baseline for every provider
configuration. The validation is minimal:

```python
required = {"kind", "model", "models"}
if all(k in data for k in required):
    if pid in DEFAULT_PROVIDERS:
        DEFAULT_PROVIDERS[pid].update(data)
    else:
        DEFAULT_PROVIDERS[pid] = data
```

Any process or agent that can write to `/data/model_catalog.json` (including the `bash`
tool â€” the workspace is separate from `/data` but an unconfined bash can `cat > /data/...`)
can inject an arbitrary provider with a malicious `base_url`, redirect all model API
calls to an attacker-controlled endpoint, and thereby exfiltrate every prompt, response,
and potentially API key value sent as a header. The `kind` field is not validated
against the allowed set (`openai`, `anthropic`, `vertex`) in this path.

**Recommendation:**
Move `model_catalog.json` validation to strictly validate `kind` (same whitelist as
`ProviderUpsert`), `base_url` (HTTPS-only, no localhost unless the same localhost-MCP
exception is intended), and sanitize `model`/`models` strings. Consider whether the
catalog file should be in `/data` at all vs. bundled as a read-only asset in the image.

### L-2 â€” `CM_MASTER_KEY` uses SHA-256 as KDF (not a password-stretching KDF)

**File:** `server.py:670â€“672`
**Severity:** LOW (documented in code; depends on key entropy)

**Description:**
```python
digest = hashlib.sha256(CM_MASTER_KEY.encode()).digest()
fkey = base64.urlsafe_b64encode(digest)
return Fernet(fkey)
```
The code comments note: "A single SHA-256 is NOT a password-stretching KDF... 
CM_MASTER_KEY must be a high-entropy random value... NOT a human-chosen passphrase."
This is a correct statement, but it is enforced only by a minimum-length check
(16 chars). A human operator who sets `CM_MASTER_KEY` to a memorable passphrase
(e.g., `my-secret-key-2026`) would have a key derivation vulnerable to offline
brute-force if `model_config.json` or `session_secret.key` is ever exfiltrated.

**Recommendation:**
Replace SHA-256 with `hashlib.scrypt` or `cryptography`'s PBKDF2 with a fixed salt
stored alongside the ciphertext (or use Fernet's own `generate_key` + key-wrapping).
The minimum-length check should be raised to 32 bytes. Add a startup warning if
`CM_MASTER_KEY` appears to have low entropy (e.g., contains only printable ASCII with
no special chars).

### L-3 â€” Feedback screenshot storage uses `report_id` as filename without randomness check

**File:** `server.py:1074â€“1096`
**Severity:** LOW

**Description:**
`_save_feedback_shot` saves a feedback screenshot as `<report_id><ext>` under
`FEEDBACK_SHOT_DIR`. If two concurrent feedback submissions with the same `report_id`
race, one could overwrite the other. More importantly, if `report_id` (which is a UUID)
is guessable or predictable, a user could craft a submission that overwrites another's
screenshot. The directory listing for admin review of feedback shots would then show
manipulated images. The UUID is generated server-side per report (`uuid.uuid4().hex`
seen elsewhere), so guessability is very low â€” this is a LOW finding rather than MED.

**Recommendation:**
Use `os.open` with `O_CREAT | O_EXCL` when writing the shot file to fail atomically
on collision, rather than silently overwriting.

### L-4 â€” `OPEN_ENROLLMENT=true` allows unrestricted self-registration

**File:** `server.py:189`, `server.py:1171â€“1172`
**Severity:** LOW (noted; expected to be off in production)

**Description:**
When `OPEN_ENROLLMENT=true`, any unauthenticated caller can register a `Member` account
via `POST /api/register`. The register endpoint returns a session token immediately
(same `must_reset` path as C-2 above). Combined with C-2, this means any internet user
can gain a `must_reset` token with zero authentication, register a TOTP authenticator,
and become a fully authenticated Member â€” all without Owner interaction.

The `fly.toml` does not set `OPEN_ENROLLMENT`, so it defaults to `false` â€” safe for
the default deploy. But there is no mechanism to prevent an operator from setting
`OPEN_ENROLLMENT=true` without realizing it enables free self-registration.

**Recommendation:**
Add a startup warning log (level WARNING, not just DEBUG) when `OPEN_ENROLLMENT=true`
is detected, making the security implication explicit in logs. Document the risk in
README/SECURITY.md.
